# 目标层（Targets Layer）设计开发规范

## 1. 核心设计原则

### 1.1 可插拔、开放式设计（输入接口）

目标层采用**完全开放式架构**，支持多科学主题、观测任务并行发展，确保系统的可扩展性和灵活性：

- **模块化设计**：每个科学主题、观测任务作为独立模块，可独立开发、测试和部署
- **插件式注册**：新增科学主题、观测任务的观测目标模型通过注册机制自动集成，无需修改核心代码
- **标准接口**：所有观测目标模型遵循统一的接口规范，确保互操作性
- **动态发现**：系统运行时自动发现和加载可用观测目标模型
- **版本管理**：支持模型版本控制和向后兼容

### 1.2 统一输出接口

无论何种目标类型、何种物理模型，都必须能够产生以下三种标准输出，确保与观测量层的无缝对接：

- **图像模型**：二维亮度分布（可包含时间/波长维度）
- **光谱模型**：波长-通量关系（可包含时间维度）
- **光变模型**：时间-通量关系（可包含波长维度）

此外，如果提供了必要的观测目标模型，还可产生并输出扩展的**偏振模型**。

## 2. 架构设计

### 2.1 分层架构

目标层采用三层架构，分离关注点：

1. **接口层 (Interface Layer)**
   - 提供准确、易用的文本接口，负责接受科学用户输入的观测目标模型，用户可显式的定义模型及参数，也可指定参考文献，并通过大模型生成参数化模型
   - 提供统一的API接口，负责与观测量层交互
   - 接收观测请求参数（坐标、波长、时间等）
   - 返回标准化的图像、光谱、光变数据
   - 实现模型注册和发现机制

2. **模型层 (Model Layer)**
   - 核心观测目标模型实现层
   - 包含各类目标物理模型（点源、延展源、活动星系核等）
   - 每个模型独立实现，遵循标准接口
   - 支持模型参数验证和边界检查

3. **数据层 (Data Layer)**
   - 负责目标输入输出模型的存储、读取和缓存
   - 管理源表（CSV/TSV格式）的解析和验证
   - 处理FITS文件的读写操作
   - 实现数据缓存机制以提高性能

### 2.2 核心组件

#### 2.2.1 模型注册器 (Model Registry)
- 动态发现和加载可用目标模型
- 维护模型名称与实现类的映射关系
- 支持模型版本管理和向后兼容

#### 2.2.2 模型工厂 (Model Factory)
- 根据模型名称和参数实例化具体模型
- 管理模型生命周期
- 提供模型配置验证

#### 2.2.3 输出标准化器 (Output Standardizer)
- 确保所有模型输出符合统一格式
- 转换数据为FITS标准格式
- 添加必要的元数据（WCS、时间戳等）
