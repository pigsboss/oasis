# 目标层（Targets Layer）设计开发规范

## 1. 需求分析

### 1.1 背景与目标

目标层的设计源于天文研究的两大核心需求：

**需求一：观测任务设计**
面向天文观测任务的设计阶段，需要将特定观测目标的参数化模型（涵盖加速、辐射、吸积、并合、瓦解、喷发等物理过程）转换为图像、光谱、光变曲线等可观测模型，以支撑观测策略优化、积分时间计算、仪器配置选择等任务规划活动。

**需求二：观测数据分析**
面向观测数据的科学分析阶段，需要从图像、光谱、光变曲线等高级科学数据产品中提取、拟合观测目标模型参数，建立数据与物理模型之间的定量联系，实现物理参数的约束和反演。

### 1.2 功能需求

基于上述背景，目标层需实现三大核心功能：

**功能一：模型全生命周期管理**
建立观测目标参数化模型的收集、组织和管理体系，包括：
- 统一接收天文用户输入的各类观测目标参数化模型
- 语义解析与歧义消除，确保模型描述的精确性
- 冲突检测机制，识别参数间物理不自洽或逻辑矛盾
- 完整性测试，验证模型是否存在未定义的物理环节或计算断点
- 版本控制与溯源，支持模型迭代历史和兼容性管理

**功能二：高效正向建模与仿真**
从参数化模型高效生成具有足够精度的可观测模型：
- 支持多尺度、多维度的物理模型数值仿真（空间、能谱、时域）
- 确保数值仿真计算精度满足观测模拟需求
- 优化计算性能，支持大规模参数空间和蒙特卡洛模拟
- 输出标准化FITS格式的图像、光谱、光变曲线数据模型，无缝衔接观测量层

**功能三：参数反演与模型检验**
从高级科学数据产品中估计模型参数并验证一致性：
- 支持从FITS标准数据产品进行模型参数提取和拟合
- 实现贝叶斯推断、最大似然估计等多种参数估计方法
- 与输入的设计阶段模型进行定量比较，计算偏离度和一致性指标
- 生成模型-数据对比报告，支撑科学结论的可靠性评估

### 1.3 科学主题与层级化组织

**科学主题（Science Themes）的概念来源**

科学主题这一组织概念具有双重来源：

1. **科学驱动**：在前期科学目标研究阶段，科学用户基于研究兴趣和方法论的自然聚类，自发形成了若干科学主题（如系外行星、多信使天文学、星系宇宙学等），这些主题代表了天文研究的前沿领域和重大问题。

2. **工程驱动**：在 OASIS 软件系统设计过程中，我们认识到若采取完全扁平化的观测目标参数化模型组织方式（即每个任务独立开发所有模型），将导致两大严重问题：
   - **重复开发**：不同任务频繁需要相同的基础模型（如研究系外行星需要恒星模型，研究多恒星系统也需要恒星模型，研究星系与超大质量黑洞互馈同样涉及恒星模型）
   - **冲突风险**：各自开发的同类模型（如恒星大气模型、吸积盘模型）在接口、参数定义、单位制等方面极易产生不一致，造成系统集成时的语义冲突和版本混乱

**层级化组织方案**

为解决上述问题，目标层采用层级化的观测目标模型组织方式，形成从通用到专用的模型继承体系：

```
基础模型（Foundation）
    ↓
科学主题（Themes）
    ↓
研究方向（Directions, Optional）
    ↓
观测任务（Missions）
    ↓
具体目标（Targets）
```

- **基础模型**：包含恒星、吸积、引力束缚态、辐射机制等跨领域通用理论模型，作为所有科学主题的共享基础设施
- **科学主题**：围绕重大科学问题组织的模型集合（如致密天体并合、行星系统形成），继承并扩展基础层模型
- **研究方向**（可选）：在大型科学主题内按方法论或对象类别划分的子领域，用于进一步细化模型管理
- **观测任务**：具体的观测提案或巡天项目，继承主题/方向的模型体系，定义特定的参数配置和观测约束
- **具体目标**：单次观测或仿真的具体天体实例，承载最终的物理参数和观测配置

此架构确保基础模型的单一来源（Single Source of Truth），通过继承和组合实现复用，同时通过命名空间隔离避免不同主题间的模型冲突。

## 2. 核心设计原则

### 2.1 可插拔、开放式设计（输入接口）

目标层采用**完全开放式架构**，支持多科学主题（方向）、观测任务（目标）并行发展，确保系统的可扩展性和灵活性：

- **模块化设计**：每个观测目标作为独立模块，可独立开发、测试和部署
- **插件式注册**：新增观测目标模型通过注册机制自动集成，无需修改核心代码
- **标准接口**：所有观测目标模型遵循统一的接口规范，确保互操作性
- **动态发现**：系统运行时自动发现和加载可用观测目标模型
- **版本管理**：支持模型版本控制和向后兼容

### 2.2 统一输出接口

无论何种目标类型、何种物理模型，都必须能够产生以下三种标准输出，确保与观测量层的无缝对接：

- **图像模型**：二维亮度分布（可包含时间/波长维度）
- **光谱模型**：波长-通量关系（可包含时间维度）
- **光变模型**：时间-通量关系（可包含波长维度）

此外，如果提供了必要的观测目标模型，还可产生并输出扩展的**偏振模型**。

## 3. 架构设计

### 3.1 总体架构

目标层采用**三段式 V 字形纵向架构**。接口段位于 V 字顶部，是 OASIS 面向用户及外部系统的唯一外部边界；模型段位于 V 字中段，是目标层的核心计算引擎；数据段位于 V 字底部，是目标层与观测量层的唯一接口。三段纵向串接，形成清晰的单向依赖关系：**接口段只应对上，数据段只对下，模型段承上启下并专注于物理计算与参数反演**。

**架构说明**

- **接口段（V 顶）**：作为 OASIS 外部接口，承担所有与用户及第三方系统的交互。输入侧通过 CLI、Python API、自然语言/LLM 接口等接收观测目标参数化模型；输出侧面向用户返回拟合参数、统计一致性评估、不确定性量化及可视化诊断报告。**不直接处理 FITS 编码或观测量层协议。**
- **模型段（V 中）**：目标层核心，内设正演与反演双引擎。正演链路将用户输入的参数化模型经冲突检测、完整性验证后，由物理计算引擎生成图像、光谱、光变、偏振等**可观测模型的内存数据结构**；反演链路接收数据段解析后的科学数据产品，执行参数估计、一致性评估与不确定性量化，产出高阶科学结果。**本段完全专注于物理模型与统计推断，不感知底层文件格式。**
- **数据段（V 底）**：目标层与观测量层之间的唯一边界。向下游接收模型段的内存对象，统一序列化为标准 FITS 格式（含 WCS、时间戳、模型版本等元数据）并输出至观测量层；向上游从观测量层读取高级科学数据产品，完成解码与验证后，交付与格式无关的标准数据对象供模型段反演。**目标层内唯有本段允许感知观测量层数据格式与通信协议。**

**数据流说明**

- **正演链路（左侧，向下）**：用户输入 → 接口段 → 模型段正演引擎生成可观测模型（内存对象）→ 数据段编码为标准 FITS → 观测量层。
- **反演链路（右侧，向上）**：观测量层高级数据产品 → 数据段解析为内存对象 → 模型段反演引擎拟合参数与一致性评估 → 接口段向用户呈现科学结论。

### 3.2 核心组件

#### 3.2.1 接口段 - 输入组件

**User Interface / 用户界面**
- 作为 OASIS 面向用户及外部系统的统一入口，承担人机交互与外部 API 协议适配
- 支持程序调用（Python API、命令行）、文件提交（YAML 参数化模型）与自然语言理解（LLM 接口）
- 仅接收用户输入，不向下游系统输出 FITS 或对接观测量层协议
- 向用户呈现任务环境资源状态与准入评估信息，支持交互式资源查询与异常告警

**Input Validator / 输入验证**
- 验证输入模型的语法合法性、字段完整性与数据类型合规
- 物理语义验证（单位制、量纲、超光速/负温度等）下放至模型段 Conflict Detector

#### 3.2.2 模型段 - 管理组件

**Hierarchy Manager / 层级管理**
- 维护基础→主题→方向→任务→目标的五层级继承链与命名空间隔离
- 处理模型版本控制与向后兼容性检查

**Conflict Detector / 冲突检测**
- 检测参数间物理不自洽与跨层级继承时的量纲冲突
- 验证单位制合法性，识别超光速、负温度等非物理取值

**Completeness Checker / 断点检测**
- 检查计算依赖图中的未定义中间量与物理环节断点
- 确保从输入参数到可观测量的转换链完整无缺

#### 3.2.3 模型段 - 正演组件

**Physics Engine / 物理引擎**
- 执行辐射、吸积、并合、瓦解等数值仿真，集成光线追踪、辐射转移、轨道演化等积分方法（含刚性方程与自适应步长）
- 管理多尺度计算（空间/能谱/时域）与自适应网格 refinement
- 产出图像、光谱、光变、偏振等可观测模型的标准内存数据结构，移交数据段序列化
- 提供标准物理常数与单位制转换服务

#### 3.2.4 模型段 - 反演组件

**Parameter Estimator / 参数估计**
- 接收数据段解析后的内存数据对象，执行贝叶斯推断（MCMC、嵌套采样）与最大似然估计
- 提供先验分布与似然函数接口，产出拟合参数及后验分布

**Statistics Evaluator / 统计评估**
- 计算模型-数据偏离度（残差、χ²、贝叶斯证据比）与统计一致性
- 估计参数误差、置信区间与可信区域，处理噪声及系统误差传播
- 提供模型选择准则（AIC、BIC、贝叶斯因子）

#### 3.2.5 模型段 - 支撑组件

**Compute Runtime / 计算运行时**
- 管理蒙特卡洛模拟与多参数空间扫描的并行执行，支持 CPU/GPU 异构调度与 MPI 分布式计算
- 实现基于参数哈希的 LRU 缓存策略，管理内存/磁盘两级缓存及失效策略，避免重复计算
- 负责任务队列、负载均衡与缓存版本一致性检查
- 内置资源监视器，归集计算、存储、网络等本地硬件资源的实时状态与可获得性评估，合并写入运行时日志
- 按用户需求向接口段提供资源状态摘要，由用户界面以友好方式呈现任务准入评估与环境概况

#### 3.2.6 接口段 - 输出组件

**Result Presenter / 结果呈现**
- 接收模型段产出的高阶科学结果，面向用户呈现拟合参数报表、统计一致性报告、不确定性量化结论与可视化诊断图表
- 支持 JSON/YAML 摘要、Markdown 报告与 Matplotlib/Plotly 图表等用户级格式
- 禁止执行 FITS 编码、WCS 构造或观测量层数据格式对接；用户下载原始数据须调用数据段服务

#### 3.2.7 数据段组件

**Observables Interface / 观测量层接口**
- 目标层与 observables 层文件级层间接口的唯一管理者，定义目录结构、命名规范、就绪信号与状态回报机制
- 正演时组织输出文件包（FITS/HDF5 主数据 + JSON/YAML/TSV/CSV 辅助文件），产出参数侧写、观测配置摘要与处理指令
- 反演时扫描并读取 observables 层放回的结果包，解析附带辅助元数据
- 探测目标层与 observables 层分布式节点间的局域网连通性与端到端延迟
- 向外部工作流编排器发送"输出就绪"或"输入可用"事件，不直接调用 observables 层内部工具

**FITS/HDF5 I/O Manager / FITS/HDF5 数据文件管理**
- 负责标准 FITS/HDF5 的序列化与反序列化（含 WCS、BINTABLE、头部关键字、时间戳与模型版本元数据）
- 支持大文件分块读写与内存映射优化，向上交付格式无关的标准内存对象

**Result Repository / 结果仓库**
- 管理正演产物与反演数据产品的本地持久化、缓存及生命周期
- 支持计算结果版本快照与缓存失效策略
- 探测存储容量、读写带宽与配额状态，支撑冷热数据分层迁移决策

### 3.3 接口定义

#### 3.3.1 外部接口（目标层对外）

外部接口承载目标层与上下游系统之间的所有正式边界，定义功能边界与性能基线，不涉及具体实现途径。

**A. 用户交互接口（由接口段承载）**

功能要求：
- **模型输入**：接收观测目标参数化模型的能力，支持显式结构化描述、自然语言语义解析结果及外部文献引用解析结果的输入
- **任务触发**：提供正演计算与反演计算的任务发起能力，支持单次任务、批量任务与参数扫描模式
- **层级检索**：提供模型层级体系（基础模型、科学主题、研究方向、观测任务、具体目标）的浏览、查询与继承链解析能力
- **结果呈现**：返回科学分析结果的能力，包括拟合参数、统计一致性评估、不确定性量化结论及可视化诊断信息
- **状态监控**：提供计算任务生命周期状态（排队、执行、完成、异常）的查询与事件通知能力
- **数据获取**：支持用户按需获取原始数据产品（正演产物或反演输入数据）的下载能力
- **资源状态呈现**：提供任务环境资源状态与准入评估的交互式查询能力

**接口文件要求（YAML 参数化模型）**

- **编码**：采用 UTF-8 编码
- **顶层必填字段**：须包含 `oasis_version`、`theme`、`target_name`、`parameters`
- **层级继承**：支持 `extends` 关键字引用基础模型或上级主题
- **物理量表达**：所有带量纲数值须以 `value` + `unit` 键值对表示，单位字符串须兼容天文标准单位制
- **参数扫描**：支持 `linspace`、`logspace`、`list` 三种参数网格定义
- **语义约束**：物理语义级校验（量纲一致性、非物理取值识别）由模型段执行，不前置在文件解析层
- **扩展预留**：当前版本以 YAML 为确定接口；若未来发现 YAML 不足以覆盖复杂结构化需求，再根据实际情况增加其它途径

性能要求：
- 交互式操作须满足低延迟响应，保证用户界面可用性
- 批量任务提交须具备高吞吐能力，支撑大规模参数空间与蒙特卡洛模拟场景
- 结果返回须支持分页、流式传输与增量更新，适应大体积诊断数据
- 资源状态查询须低延迟反馈，避免阻塞用户交互

**B. 观测量层交换接口（由数据段承载）**

功能要求：
- **正演输出**：向观测量层输出可观测模型数据产品的能力，包括物理模型数据、坐标参考信息、时频域描述及必要的辅助参数侧写
- **反演输入**：从观测量层接收高级科学数据产品的能力，包括观测数据、噪声模型、仪器响应信息、数据质量标记及校准元数据
- **完整性校验**：支持层间数据包完整性与版本兼容性校验的能力，确保上下游对数据契约的理解一致
- **状态同步**：支持异步处理状态同步的能力，包括数据就绪、处理中、完成及异常回退等状态的双向感知
- **辅助信息交换**：支持处理指令、观测配置参考及任务标识等辅助元数据的交换能力

**接口文件要求（FITS 层间数据包）**

- **正演输出（目标层 → observables 层）**
  - 主数据文件：采用标准 FITS，封装图像模型（2D/3D/4D 亮度分布）、光谱模型（波长-通量关系）、光变模型（时间-通量关系）及可选偏振模型
  - WCS 要求：图像与光谱数据须包含完整、自洽的 WCS 描述
  - 头部关键字：须包含 `OASIS_VER`、`MODEL_ID`、`TIMESTAMP`、`TARGET_CLASS` 及 `OASIS_OBS_INTF_VER`
  - 辅助文件包：配套 JSON/YAML/TSV/CSV 辅助文件，携带参数侧写、观测配置摘要与处理指令
  - 就绪信号：数据段完成写入后，在层间交换目录生成 `.forward_ready` 哨兵文件

- **反演输入（observables 层 → 目标层）**
  - 数据产品文件：采用标准 FITS，由 observables 层生成的高级科学数据产品
  - 头部元数据：须包含 `TASK_ID`、`INSTRUMENT`、`CALIB_VER`、`DATA_QUAL`
  - 辅助文件包：配套 JSON/YAML 辅助文件，声明观测参数、噪声模型、校准系数及数据质量说明
  - 就绪信号：observables 层完成处理后，在交换目录生成 `.inverse_ready` 哨兵文件

- **错误回退**：若 observables 层处理失败，须在交换目录放置 `.error` 文件，内含错误码与英文错误摘要

- **扩展预留**：当前版本以 FITS 为确定接口；若未来发现 FITS 不足以覆盖超大体积、层次化多维数组等需求，再根据实际情况增加其它途径（如 HDF5）

性能要求：
- 支持大体积科学数据的高效传输，满足天文典型数据规模需求
- 支持并发任务的严格数据隔离，避免任务间交叉污染
- 序列化与反序列化性能须匹配观测模拟与数据分析的实时性要求
- 支持异常场景下的容错与重试机制，保障长链路任务可靠性

**C. 本地硬件接口（由 Compute Runtime 统一归集，相关子系统分项承载）**

功能要求：
- **网络连通性**：探测互联网可达性（外部天文数据库、文献检索服务、模型下载源），保障用户输入阶段的外部资源获取；在目标层、观测量层、载荷层分布式部署时，探测局域网层间通路的可用性与端到端延迟
- **计算资源**：评估 CPU/GPU/分布式节点的可用规模与实时负载，匹配正演数值仿真与反演统计推断的资源需求；支持异构环境的任务调度、资源预留与释放
- **存储资源**：评估本地及分布式存储的可用容量、读写带宽与配额；支撑大体积科学数据（正演产物、反演输入、缓存文件）的高吞吐持久化、分层生命周期管理与海量数据可靠存储
- **状态报告与降级**：向用户或管理员报告网络拓扑、资源瓶颈及异常状态；在资源不足或网络异常时触发排队、拒绝或降级策略，避免阻塞观测任务时效性

性能要求：
- 探测与状态采集须低开销，避免挤占科学计算与数据传输带宽
- 层间网络须满足天文典型数据规模传输的带宽与时延基线，支持大文件高效交换
- 计算资源须提供低延迟的负载评估与弹性调度，适应蒙特卡洛并发与参数扫描的资源波动
- 存储须匹配 GB–TB 级单文件的高吞吐连续读写与并发 I/O 隔离，具备冗余容错能力防止数据丢失

#### 3.3.2 内部接口（目标层内）

内部接口遵循**段间内存对象契约**原则：各段之间一律通过与文件格式无关的标准抽象数据交互，禁止跨段直接操作底层字节流或原始文件句柄。

**A. 输入管道接口（接口段 → 模型段）**

功能要求：
- 传递结构化观测目标参数化模型的能力，承载层级定位、物理参数、单位制声明、观测约束与先验信息
- 传递用户意图与任务上下文的能力，支持正演/反演模式标识及计算精度要求
- 支持模型版本与继承链定位的能力，确保模型段正确解析基础模型引用与主题扩展关系

性能要求：
- 支持复杂嵌套参数结构的低延迟、低损耗传递
- 支持大参数空间模型对象的高效传输，避免冗余拷贝

**B. 正演管道接口（模型段 → 数据段）**

功能要求：
- 传递标准化可观测模型内存对象的能力，涵盖图像、光谱、光变、偏振及其多维组合
- 携带物理单位、坐标网格、波长/时间轴及模型生成元数据的能力
- 支持多尺度、多维度数据描述的能力，适应自适应网格与非规则采样场景

性能要求：
- 支持大规模数值数组的零拷贝或低拷贝传输
- 支持分块数据与懒加载场景的流式传递，降低内存峰值占用

**C. 反演管道接口（数据段 → 模型段）**

功能要求：
- 传递解析后科学数据产品的能力，将底层天文数据格式完全解码为与格式无关的标准内存表示
- 传递观测元数据、噪声模型及数据质量标记的能力
- 传递仪器配置与校准信息的能力
- 严格屏蔽底层文件格式细节，向上游仅暴露物理量与标准坐标对象

性能要求：
- 支持大数组数据的内存高效表示，支撑反演算法的频繁随机访问
- 支持坐标变换后的快速索引与切片，满足统计推断的计算密度要求

**D. 结果管道接口（模型段 → 接口段）**

功能要求：
- 传递高阶科学结果的能力，包括拟合参数、后验分布摘要、偏离度指标与模型选择统计量
- 传递模型评估与不确定性量化结论的能力
- 传递诊断数据与模型-数据对比指标的能力，支撑接口段生成用户级报告

性能要求：
- 支持统计结果集的快速聚合与按需裁剪
- 支持后验分布等大型中间结果的分块或流式传递，避免接口段内存压力

**E. 计算支撑接口（模型段 ↔ 计算运行时）**

功能要求：
- 提交数值计算任务描述的能力，包括参数哈希、计算图、精度要求与资源需求
- 获取计算资源调度状态与缓存命中信息的能力
- 支持蒙特卡洛并行、参数扫描及异构计算协作的能力

性能要求：
- 支持高并发计算任务的低延迟调度与结果回收
- 支持计算缓存的快速命中检测与失效同步，避免重复数值计算

**F. 持久化支撑接口（数据段 ↔ 结果仓库）**

功能要求：
- 正演产物与反演数据产品的存储与检索能力
- 计算缓存与版本快照的管理能力
- 层间交换数据的临时托管与生命周期管理能力

性能要求：
- 支持大体积数据产品的快速写入与读取
- 支持缓存失效策略的高效执行与存储空间回收

**G. 数据服务接口（接口段 → 数据段，受控横向）**

功能要求：
- 支持接口段按任务标识获取原始数据产品，供用户下载
- 支持查询任务相关数据产品状态与元数据的能力
- 严格限定为**用户级数据服务**，禁止绕过模型段直接进行科学计算相关的数据交换

性能要求：
- 支持大文件流式传输，降低接口段内存占用
- 支持数据访问权限控制与生命周期管理，防止未授权访问

**H. 资源状态接口（模型段 → 接口段）**

功能要求：
- 传递本地硬件资源实时状态与任务准入评估的能力，涵盖计算负载、存储余量与带宽、层间网络延迟
- 支持按需查询与任务级事件驱动推送两种模式
- 提供资源不足、网络异常等准入拒绝或降级建议的传递能力

性能要求：
- 资源状态采集与传递须低开销，避免挤占主计算链路
- 状态同步须满足交互式查询的低延迟要求，保障用户界面反馈及时性
