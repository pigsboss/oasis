# 采用架构决策记录机制

- **状态**：已接受
- **日期**：2026-04-24
- **决策者**：OASIS 目标层设计团队
- **涉及文档**：docs/design/adr/README.md, docs/design/adr/template.md

## 背景
OASIS 目标层是一个多科学主题、多任务类型的复杂科学计算系统。在早期设计阶段，我们做出了许多影响深远的架构选择，例如分层、数据类型、接口契约等。这些决策背后的原因、权衡和替代方案若不记录下来，随着时间推移和团队变动，将难以被后续维护者和新成员理解，导致"为什么系统长这样"的知识丢失，增加未来演进的风险。

## 决策
采用轻量级架构决策记录（ADR）机制，在 `docs/design/adr/` 目录下，以 Markdown 文件形式记录所有重要架构决策。每条 ADR 包含背景、决策、理由、影响和替代方案等标准章节。新建 ADR 遵循标准模板，并通过 Pull Request 与代码/文档变更一同评审。

## 理由
- **可追溯性**：将隐性知识显性化，使未来的开发者能够理解历史决策的上下文。
- **轻量级**：采用 Markdown 平面文件，与现有文档体系（docs/）融合，无需额外工具。
- **标准化**：经典 ADR 格式在工业界广泛使用，团队容易接受和学习。
- **与设计文档互补**：设计规范（如 targets.md）描述"是什么"，ADR 记录"为什么"。

## 影响
- 所有后续重大设计变更必须伴随 ADR 的创建或更新。
- 在代码评审中，涉及核心架构的 PR 需检查 ADR 是否存在。
- 维护者需要花费少量额外时间编写和评审 ADR，但长期收益远大于成本。

## 考虑的替代方案
- **方案1**：将所有决策记录在 wiki 或 Confluence 中
   - 优点：易编辑，支持富文本。
   - 缺点：与代码仓库分离，版本控制弱，不易随代码同步演进。
- **方案2**：不做结构化记录，依赖代码注释和历史邮件
   - 优点：零初始成本。
   - 缺点：知识碎片化，不可维护，新人上手困难。
