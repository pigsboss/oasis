# 架构决策记录 (Architecture Decision Records, ADR)

本目录归档 OASIS 目标层及其他子系统在设计与演化过程中的重要架构决策。每一条决策记录提供**背景、决策内容、理由、影响**以及**考虑过的替代方案**，帮助未来维护者理解系统为什么是现在的样子，而非仅仅查看“是什么”。

## 目录结构

```
docs/design/adr/
├── README.md          # 本文件
├── template.md        # 新决策记录的模板
├── 0001-record-architecture-decisions.md
├── 0002-use-target-histogram-for-distributions.md
├── 0003-introduce-transmission-model.md
└── ...
```

文件命名规则：`NNNN-简短英文描述.md`，序号为四位数字，从前向后递增。英文描述使用小写与连字符。

## 模板

新建 ADR 时请复制 `template.md` 并按以下结构填写：

```markdown
# [标题]

- **状态**：[提议中 | 已接受 | 已弃用 | 已替代]
- **日期**：YYYY-MM-DD
- **决策者**：[姓名或角色]
- **涉及文档**：[相关设计文档的相对路径，如 docs/guides/targets.md]

## 背景
[什么需求或问题促成了此决策？提供必要的上下文和约束。]

## 决策
[我们决定做什么。描述清晰、无歧义。]

## 理由
[为什么做出这个决策？技术、性能、可维护性、团队协作等方面的理由。]

## 影响
[该决策对系统架构、组件、接口、文件格式、流程等的影响；需要调整的其他文档或模块。]

## 考虑的替代方案
- **方案1**：[简述]
   - 优点：
   - 缺点：
- **方案2**：
   - 优点：
   - 缺点：
```

> **注意**：模板中的注释和占位标记在实际文档中应删除或替换。

## 状态说明

| 状态 | 含义 |
|------|------|
| `提议中` | 决策已被提出，正在讨论和评审，尚未正式生效。 |
| `已接受` | 决策已评审通过并在实际设计中执行。 |
| `已弃用` | 该决策已被后续决策取代，不再适用，但保留历史参考。 |
| `已替代` | 同“已弃用”，可注明被哪条新决策替代。 |

## 何时编写 ADR

满足以下任一条件时，应当创建一条 ADR：

- 引入或删除一个核心数据类型、抽象基类或关键设计模式。
- 对架构分层、边界或接口原则做出重大调整。
- 采用或弃用一项外部依赖或技术协议（如 FITS 作为唯一层间格式）。
- 拒绝了一个被广泛讨论的替代方案。
- 命名约定、模型关系等概念发生重大变化，可能影响下游理解。

日常功能增强（如增加一个新光谱子类）或小范围重构不需要 ADR，可通过代码提交记录与常规文档更新说明。

## 与设计文档的关系

- **`docs/guides/targets.md`** 等设计规范文件描述系统“是什么”和“怎么做”。
- **ADR** 记录“为什么这样设计”，为设计规范提供历史上下文和决策依据。
- 设计文档中若出现重大决策点，可添加 ADR 引用（例如“关于为何采用三段式架构，见 ADR-0001”）。
- ADR 不会被用来替代规范，也不会因设计细微迭代而频繁更新；它是相对稳定的记录。

## 使用流程

1. 在分支上拷贝 `template.md` 并重命名为 `NNNN-<slug>.md`，填写内容。
2. 通过 Pull Request 提交 ADR，同时包含该决策涉及的代码或文档变更。
3. 评审者验证决策的合理性、完整性和一致性。
4. 当 ADR 被合并到主分支后，将其状态设为 `已接受`。
5. 若未来某个决策被新决策取代，修改原 ADR 的状态为 `已弃用` 或 `已替代`，并在新 ADR 中注明替代关系。
