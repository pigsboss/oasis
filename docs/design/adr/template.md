# ADR 001: TargetModel 基类字段设计

- **状态**：已接受
- **日期**：2026-04-04
- **决策者**：OASIS 架构团队
- **涉及文档**：docs/design/targets_data_model.md

## 背景

目标层需要为所有参数化模型（`SourceModel`、`BackgroundModel`、`TransmissionModel`）与可观测模型（`TargetHistogram`）提供统一的基类，以承载模型实例的通用标识、溯源信息、版本控制及层级归属。在 `targets_data_model.md` 中已经概述了基类 `TargetModel` 的设计目标与字段定义，需要将其正式确定为架构决策。

主要驱动力：

- **唯一标识与可引用性**：模型实例之间需要稳定的逻辑标识，以支持跨模型引用（例如直方图追溯其源模型）。
- **溯源与科学可重复性**：必须明确记录模型的来源（用户、文献、算法或反演更新）与版本，支持结果复现与审计。
- **层级组织协作**：模型需要与 OASIS 的五级模型层级（基础 → 主题 → 方向 → 任务 → 目标）协同，提供归属定位信息。
- **可扩展性**：通过 `metadata` 字典携带额外描述信息，避免核心接口膨胀。
- **生态兼容性**：字段类型优先采用 Python 标准类型与 Astropy 生态类型，确保科学计算栈的互操作。

## 决策

我们决定采用 `targets_data_model.md` 中定义的 `TargetModel` 基类字段设计，具体如下：

- `name`（`str`，必需）：模型实例的逻辑名称，在所属层级命名空间内唯一。
- `provider`（`str`，必需）：模型提供者标识，默认值 `"user"`。
- `version`（`str`，必需）：语义化版本号，推荐 `MAJOR.MINOR.PATCH` 格式。
- `metadata`（`dict[str, Any]`，可选）：用户自定义扩展字典，默认为空字典；禁止存放影响数值计算的物理参数。
- `hierarchy`（`Optional[HierarchyDescriptor]`，可选）：模型在五级层级体系中的定位信息，由 `Hierarchy Manager` 在解析/注册时填充，初始为 `None`。

辅助类型 `HierarchyDescriptor` 的字段（`theme`, `direction`, `mission`, `target`）作为接口概要已在数据模型文档中给出，其完整实现在层级管理器的详细设计中另行定义。

## 理由

1. **唯一标识与可引用性**：`name` 和 `hierarchy` 的组合可以唯一确定模型实例，满足模型间引用需求。
2. **溯源与可重复性**：`provider` 与 `version` 共同记录模型的来源与迭代历史，支持完整的溯源链。
3. **层级协作**：`hierarchy` 字段允许模型记录其归属层级，与 OASIS 的五级组织架构直接对应。
4. **可扩展性**：`metadata` 字典允许用户存储额外描述信息，而不需要在基类中增加大量可选字段，保持了核心接口的简洁。
5. **生态兼容性**：字段类型均为 Python 标准类型或可序列化类型（`dict`），便于与 `astropy` 等库的 `Quantity`、`SkyCoord` 等自定义类型协同使用（这些具体类型的值应存放在子类中，而非基类）。
6. **一致性**：所有模型子类共享相同的元数据协议，降低了集成成本和认知负担。

## 影响

- **模型段**：`TargetModel` 的 `hierarchy` 字段将在 `Hierarchy Manager` 注册模型时自动填充；现有模型基类需按照此定义重构。
- **接口段**：输入验证器需检查 `name`、`provider`、`version` 的格式合法性；用户提交的模型 YAML 可以省略 `provider`（使用默认值）和 `metadata`（使用空字典）。
- **数据段**：序列化 / 反序列化时，`hierarchy` 需要作为可选的描述性元数据写入 FITS 头或辅助 JSON 文件中。
- **可维护性**：`metadata` 字典的滥用风险需通过代码审查和自动化检查（禁止物理参数放入）来管控。
- **文档**：`targets_data_model.md` 的现有定义已足够，无需立即修改；后续 `HierarchyDescriptor` 的详细实现在层级管理器的设计文档中补充。

## 考虑的替代方案

- **方案1：将所有字段扁平化为基类必填字段**（例如将 `hierarchy` 拆分为 `theme`, `direction`, `mission`, `target` 四个必填字段）
   - 优点：不需要引入辅助类型，直接存在于基类中，易于序列化。
   - 缺点：层次结构不灵活（例如方向或任务可以为空时需使用空字符串或占位符），与可选层级设计冲突；基类接口膨胀。
   - **否决**：因为 OASIS 的层级是可选的（支持直接从主题跳到目标），可选化的字段更自然。

- **方案2：使用 `PlainDict` 或 `SimpleNamespace` 作为基类，不限制类型**
   - 优点：完全灵活，不需要预先定义字段。
   - 缺点：失去类型安全；不利于静态分析与 IDE 提示；不利于版本控制与向前兼容。
   - **否决**：因系统需要严格的接口规范。

- **方案3：基类只包含 `name` 和 `version`，其他信息通过联合表或外部注册表管理**
   - 优点：基类极小化。
   - 缺点：引入额外查询开销，不利于内存效率；跨模块引用时需要额外的查找，增加复杂度。
   - **否决**：因为模型中频繁需要直接访问 `provider` 和 `metadata`，就地存储更高效。
