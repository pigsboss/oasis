# 建立核心参数化模型层次结构

- **状态**：已接受
- **日期**：2026-04-24
- **决策者**：OASIS 目标层设计团队
- **涉及文档**：docs/guides/targets.md, docs/diagrams/targets/classes.puml

## 背景
目标层的核心使命是处理观测目标的参数化模型。天文领域模型种类繁多，包括恒星、星系、吸积盘、潮汐瓦解事件等。我们需要一个统一且可扩展的模型层次，既能捕获不同天体类型和辐射过程的共性（坐标、自行、光谱），又能方便地为特定科学主题（如系外行星、活动星系核）定制专用模型。同时，我们需要一个清晰的"信号 + 背景 + 消光"组合范式，以支持正演合成和反演分离。

## 决策
引入以 `TargetModel` 为抽象基类的类层次结构，定义三种具体参数化模型类型：
- `SourceModel`：信号源模型，包含天体测量坐标、自行和辐射光谱描述。
- `BackgroundModel`：背景辐射模型，包含空间区域和光谱描述。
- `TransmissionModel`：前景消光/透过率模型，包含光学深度和透过率曲线。

所有模型继承自 `TargetModel`，后者提供名称、来源引用、元数据和版本等公共字段，并携带层级属性（基础→主题→方向→任务→目标），支持 `Hierarchy Manager` 进行继承链管理。光谱描述统一抽象为 `SpectralModel`，具体类型（黑体、幂律等）由子类实现，基类仅定义接口。

此外，采用 `astropy.coordinates.SkyCoord` 和 `astropy.units.Quantity` 作为坐标和单位的表示标准，但保持松耦合，仅要求兼容性，不强依赖。

## 理由
- **统一性与复用**：公共属性提升到基类，避免各个子模型重复定义名称、版本等。
- **灵活组合**：正演引擎可以接受信号、背景、透过率的任意组合，覆盖从简单点源到复杂复合场景。
- **关注点分离**：背景和透过率独立建模，用户可分别定义、测试和复用，例如多个信号源共用同一背景或消光谱。
- **类型安全与扩展**：通过继承，每种模型可扩展额外的物理参数（如恒星的质量、金属丰度），同时保持核心接口一致。
- **与 astropy 兼容**：利用社区广泛认可的天文计算标准，避免重新发明坐标系和单位系统，降低学习曲线，提升互换性。

## 影响
- 类图（`classes.puml`）将体现该继承和组合关系。
- 所有输入解析（YAML）、冲突检测、完整性检查组件都必须支持这些新模型类型。
- 反演输出将返回更新后的 `SourceModel`、`BackgroundModel` 和 `TransmissionModel` 实例。
- 未来添加新辐射模型（如康普顿散射）只需扩展 `SpectralModel` 子类，不影响现有框架。

## 考虑的替代方案
- **方案1**：使用单一 `ParameterSet` 字典，不区分信号/背景/透过率。
   - 优点：简单，灵活性最高。
   - 缺点：失去类型安全，不同组成部分的语义混淆，冲突检测困难，难以从数据中分离各分量。
- **方案2**：为每类天体（恒星、星系、AGN）创建完全独立的模型类，无公共基类。
   - 优点：高度定制化。
   - 缺点：大量重复代码，管理混乱，难以实现通用处理管道。
- **方案3**：基类命名为 `OasisModel`。
   - 优点：带有项目标识。
   - 缺点：名称不直接表达其作为参数化目标模型的本质，易与系统其它概念混淆。最终选择 `TargetModel` 以强调"观测目标"这一核心概念。
